1. Decisions (Based on Documentation)

1.1 Local vs Strapi Cloud

Decision: Set up Strapi locally for development.

  • Why local: Easier to connect to the existing Next.js project (mosc-temp at E:\project_workspace\mosc-temp), full control over schema and lifecycle hooks, no vendor dependency during build, and fits the “budget-friendly / self-hosted” recommendation in the spec.
  • Strapi Cloud later: You can deploy the same codebase to Strapi Cloud for production if you want managed hosting, or self-host (VPS/Docker) for lowest cost.

1.2 Use Editorial Template or Start from Scratch?

Decision: Prefer cloning the official Strapi Cloud template repo locally (easiest); otherwise create from scratch with create-strapi-app and add content types manually.

  • The spec states the editorial template “saves initial setup time (hours to days)” and “gives a common editorial baseline.”
  • On Strapi Cloud, when you click “Use template” you get a Blog (editorial-style) template. The same project is available as a public Git repo — you can clone it and run it locally without using Strapi Cloud.
  • Recommended path: Clone strapi/strapi-cloud-template-blog into cms/ in this repo, then extend it with Tenant, Sidebar Block, Ad Slot, and i18n. See §1.3 and Phase 1 (Alternative) below.

1.3 Replicate Strapi Cloud Editorial/Blog Template from Git — Is It a Good Approach?

Yes. Cloning the Strapi Cloud template from GitHub is a good approach and the easier way to get a solid, industry-standard start.

  • What you get: The same pre-configured Strapi project that Strapi Cloud offers when you “Use template” (Blog/editorial). It includes predefined content types (e.g. Article, Category), config, database migrations, sample data scripts, and a modern Strapi version (e.g. Strapi 5.x) with the latest rich-text editor and admin.
  • Rich text & features: Strapi 5 ships with an improved rich-text editor and built-in features. The template gives you a working schema and seed data so you don’t build Article/Category from scratch. Internationalization (i18n) is built into Strapi — you enable it in Admin (Settings → Internationalization) and add locales; the template may or may not enable i18n by default, but turning it on is straightforward.
  • Easier way: Instead of running create-strapi-app and then manually creating every content type, clone the repo once, install dependencies, and run. Then add only what’s missing for your spec: Tenant, Sidebar Promotional Block, Advertisement Slot, and enable i18n if not already on.

Official repo: https://github.com/strapi/strapi-cloud-template-blog. It is public and maintained by Strapi. You can clone it into your repo (e.g. as cms/) and develop locally; later you can deploy the same codebase to Strapi Cloud or self-host.

Easier way in one sentence: Clone strapi/strapi-cloud-template-blog into cms/, run npm install and npm run develop, then add Tenant, Sidebar Block, Ad Slot, and i18n — instead of creating Article/Category and the rest from scratch.

2. Prerequisites

  • Node.js 18.x or 20.x (LTS). Check: node -v
  • npm 8+ or yarn. Check: npm -v or yarn -v
  • Database: SQLite (default, good for local dev) or PostgreSQL / MySQL if you prefer. Strapi will prompt during create-strapi-app.
  • Git: This project is in catholicatenews-in-temp; the Next.js front end is in E:\project_workspace\mosc-temp.

3. Phase 1 — Create Strapi Project

Step 1.1 — Choose project location

Create the CMS inside this repo so backend and docs stay together. From the repo root:

cd E:\project_workspace\catholicatenews-in-temp
mkdir cms
cd cms

Step 1.2 — Create Strapi app

Run the official quickstart (interactive prompts for DB and project name):
npx create-strapi-app@latest . --quickstart

Or with a specific database (e.g. PostgreSQL) later you can run without --quickstart and choose options. For the fastest start, --quickstart uses SQLite and opens the admin registration in the browser.

Step 1.3 — First run

  • When the process finishes, Strapi will start. Create the first admin user in the browser (e.g. http://localhost:1337/admin).
  • Stop the server (Ctrl+C) when done. We will add content types next.

3 (Alt). Phase 1 — Clone Strapi Cloud Template (Easier Start)

Use this path if you want the official Strapi Cloud Blog template locally with minimal setup.

Recommended: Clone into a different folder (avoid Git conflicts)

If this project is already a Git repo (e.g. catholicatenews-in-temp), cloning the template into a separate folder and copying your documentation there is both possible and wise:

  • No Git conflict: The template has its own .git and history. Cloning it inside your repo would create a nested repo; cloning it outside keeps two clean repos.
  • Steps: Clone strapi-cloud-template-blog into a new folder (e.g. E:\project_workspace\catholicatenews-cms), then copy your documentation/ folder (and any other files you want) from catholicatenews-in-temp into that new project. The CMS project then contains the template + your specs and next-steps docs.
  • Result: One repo = CMS (template + your docs + your future Strapi customizations). Your original repo can stay as-is or be archived. The Next.js front end (mosc-temp) points at this CMS repo when running locally or at its deployed URL.

See § “Clone in a separate folder (two-repo setup)” below for exact commands.

Step A.1 — Clone the template

Option 1 — Clone into a different folder (recommended when current project is already a Git repo):

# From anywhere, e.g. project_workspace:
cd E:\project_workspace
git clone https://github.com/strapi/strapi-cloud-template-blog.git catholicatenews-cms
cd catholicatenews-cms

Then copy your documentation from the existing project:

# Create documentation folder and copy your docs (run from catholicatenews-cms):
mkdir documentation
copy E:\project_workspace\catholicatenews-in-temp\documentation\*.html documentation\
copy E:\project_workspace\catholicatenews-in-temp\documentation\*.png documentation\

Option 2 — Clone into cms/ inside this repo (only if you want CMS and docs in one repo and are okay with a nested Git clone or will remove cms/.git and commit the template files into the parent repo):

# From repo root (catholicatenews-in-temp):
git clone https://github.com/strapi/strapi-cloud-template-blog.git cms
cd cms

If you use Option 2 and want a single repo (no nested clone), after cloning run Remove-Item -Recurse -Force cms\.git (PowerShell) so cms/ is just files; then git add cms and commit. You lose easy git pull from the Strapi template.

Step A.2 — Install dependencies

npm install
# or: yarn install

Step A.3 — Optional: remove Strapi Cloud plugin for local-only use

The template includes @strapi/plugin-cloud for Strapi Cloud deployment. For local development you can leave it (it won’t affect local run) or remove it to avoid Cloud prompts:

npm uninstall @strapi/plugin-cloud

If you remove it, delete or comment out its registration in src/admin/app.js (or the config file where plugins are loaded) if present.

Step A.4 — First run

npm run develop
# or: yarn develop

Strapi will start (typically http://localhost:1337). Create the first admin user in the browser. The template usually includes seed scripts (e.g. npm run seed:example) — run them if you want sample articles/categories.

Step A.5 — What you have out of the box

  • Predefined content types (e.g. Article, Category) and schema
  • Strapi 5 with the current rich-text editor and admin
  • Config, database migrations, and optional seed data

Then follow Phase 2 to add only what’s missing: Homepage Layout (single type), Sidebar Promotional Block, Advertisement Slot, Tenant, and enable i18n in Settings → Internationalization. You do not need to recreate Article/Category from scratch.

Clone in a separate folder (two-repo setup)

When your current project is already a Git repo, cloning the template into a different folder avoids conflicts:

  1. Clone the template: git clone https://github.com/strapi/strapi-cloud-template-blog.git catholicatenews-cms (from E:\project_workspace or any parent folder).
  2. Copy your documentation: create documentation/ in catholicatenews-cms and copy init_requirements_spec.html, cms_setup_next_steps.html, and any images (e.g. catholicatenews_in_full_screen_shot.png) from catholicatenews-in-temp\documentation\.
  3. Commit the docs in the new repo: cd catholicatenews-cms then git add documentation, git commit -m "Add project documentation". The CMS project is now self-contained with template + your specs.

Your original repo (catholicatenews-in-temp) can stay as-is or be archived. The Next.js front end (mosc-temp) will use this CMS project’s URL (local or deployed).

4. Phase 2 — Bootstrap with Editorial Template (or Equivalent)

If you used Phase 1 (Alternative) and cloned the Strapi Cloud template, you already have Article and Category. Add only: Homepage Layout, Sidebar Block, and (in Phase 3) Ad Slot and Tenant. If you started from create-strapi-app (Phase 1), create all content types below.

Step 2.1 — Content-Type Builder (Admin UI)

Start Strapi: from cms/ run npm run develop. In Admin → Content-Type Builder:

  • If you cloned the Strapi Cloud template: Article and Category already exist. Create only: Homepage Layout (single type), Sidebar Promotional Block (single type), and in Phase 3 add Advertisement Slot and Tenant.
  • If you started from create-strapi-app: Create all of the following.

Collection: Category

Skip if you cloned the Strapi Cloud template — Category already exists.

  • name (Text, short)
  • slug (UID, attached to name)
  • description (Text, long) — optional

Collection: Article

Skip if you cloned the Strapi Cloud template — Article already exists. If needed, add a views (Integer) field for “most read” and enable i18n in Settings → Internationalization.

  • title (Text)
  • slug (UID, attached to title)
  • content (Rich text)
  • excerpt (Text, long) — optional
  • coverImage (Media, single image)
  • category (Relation: many Articles → one Category)
  • publishedAt (Date) — use Strapi’s built-in draft/publish
  • views (Integer, default 0) — for “most read” ordering (add this to the cloned template’s Article if missing)

Enable i18n on Article (and Category if you want per-locale categories) in Settings → Internationalization.

Single Type: Homepage Layout (or “Homepage”)

  • title (Text) — e.g. “Home”
  • heroBanner (Media, single image)
  • mostReadSectionTitle (Text)
  • mostReadSectionBanner (Media, single image)
  • mostReadMaxItems (Integer, default 5)

Single Type: Sidebar Promotional Block

  • blockType (Enumeration: video, social, banner)
  • thumbnail (Media, single image)
  • videoUrl (Text) — or embed URL
  • embedCode (Text, long) — for iframe/embed HTML
  • moreVideosLink (Text) — URL
  • rotateWithOthers (Boolean) — optional, for rotation

Restart Strapi after creating types so API routes are generated.

5. Phase 3 — Add Project-Specific Content Types & Features

Step 3.1 — Advertisement Slot (repeatable / collection or component)

Create a collection Advertisement Slot (or a reusable component used in a “Layout” single type) with:

  • position (Enumeration: top, between_sections, sidebar)
  • media (Media) or embedHtml (Text, long)
  • startDate / endDate (DateTime) — active range
  • priority (Integer) — for ordering/rotation

Step 3.2 — Most-read logic

  • Articles already have views. Expose a public or internal API from Next.js (or Strapi custom controller) to increment view count when a user opens an article.
  • In Next.js, fetch “most read” with: GET /api/articles?sort=views:desc&pagination[limit]=5 (and tenant/locale filters — see Phase 4).

Step 3.3 — Lifecycle hooks (optional but recommended)

Use Strapi lifecycle hooks to auto-set publishedAt, validate relations, or (later) auto-assign tenant. Example location: src/api/article/content-types/article/lifecycles.js.

6. Phase 4 — Multi-Tenant, i18n, Permissions

Step 4.1 — Tenant (multi-tenant)

Per the spec, use Pattern 1: Single Strapi + Tenant field.

  • Create collection Tenant: name, slug (UID), domain (Text, optional).
  • Add relation tenant (many Articles → one Tenant) to Article. Also add tenant to Sidebar Promotional Block, Homepage, and Advertisement Slot if they are per-tenant.
  • In Strapi Admin, create a tenant (e.g. slug mosc). Assign it to content entries.
  • Use lifecycle hooks to auto-assign tenant from the current user’s context if you have per-tenant roles.

Tenant assignment policy (no manual tenant choice)

Super Admin assigns a tenant to each user once via Editor Tenant Assignment (map admin user email → Tenant). After that:

  • Any content type created by that user (Article, Advertisement Slot, etc.) must use that tenant by default. There is no need for the user to manually assign or choose a tenant ID.
  • The tenant field must not be displayed in the Content Manager for Editors (or any non–Super Admin). The tenant field must not appear when creating or editing content — it is always set automatically from the user's assignment.

See documentation/user_guide/admin_user_guide.html for step-by-step Editor Tenant Assignment and role configuration.

Step 4.2 — API filtering by tenant

Next.js (or API consumers) must always filter by tenant, e.g.:

GET /api/articles?filters[tenant][slug]=mosc
GET /api/sidebar-promotional-block (single type — ensure it’s tenant-scoped in your design or use a collection per tenant)

For Single Types, you may need a custom controller or a “Homepage” per tenant (convert to collection “Homepage” with relation to Tenant) so that each tenant has its own homepage and sidebar block.

Step 4.3 — Internationalization (i18n)

  • Settings → Internationalization: add locales (e.g. English, Malayalam).
  • Enable i18n on Article, Category (if desired), and any other content types that need per-language content.
  • Next.js: use next-intl or next-i18next and locale-aware routes (/en/news/..., /ml/news/...). Fetch with ?locale=en or ?locale=ml in Strapi API calls.

Step 4.4 — Roles & permissions

  • Settings → Users & Permissions: create roles (e.g. Editor, Publisher, Admin).
  • Grant permissions per content type (create, read, update, delete) and restrict public API so only needed endpoints are public (e.g. find, findOne for Article with tenant filter).
  • Optionally use “Pattern 2” (per-tenant roles) so editors only see their tenant’s content (customize admin or use Strapi’s role-based visibility).

7. Phase 5 — Connect Next.js (mosc-temp)

Step 5.1 — Strapi URL

When Strapi runs locally it is usually at http://localhost:1337. In production, use your Strapi Cloud or self-hosted URL.

Step 5.2 — Environment variable in mosc-temp

In E:\project_workspace\mosc-temp, add to .env.local:

NEXT_PUBLIC_STRAPI_URL=http://localhost:1337
# or for server-side only:
STRAPI_URL=http://localhost:1337

Step 5.3 — Fetch content in Next.js

  • Resolve tenant (e.g. from hostname or path: mosc.example.com → mosc).
  • Fetch homepage, articles, most-read, sidebar block from Strapi REST (or GraphQL) with filters[tenant][slug]=<tenant> and locale=<locale>.
  • Example: fetch(`${STRAPI_URL}/api/articles?filters[tenant][slug]=mosc&sort=views:desc&pagination[limit]=5&locale=en`).

Step 5.4 — CORS

In Strapi config/middlewares.js (or middlewares.ts), allow your Next.js origin (e.g. http://localhost:3000) so the browser can call the API.

9. Summary: Order of Operations

  1. Install Strapi locally in cms/ inside catholicatenews-in-temp — either by cloning strapi/strapi-cloud-template-blog (recommended, see Phase 1 Alternative) or by running create-strapi-app.
  2. Bootstrap with an editorial-like schema: if you cloned the template, add only Homepage, Sidebar Block, Ad Slot, Tenant; otherwise create Article, Category, Homepage, Sidebar Block, Ad Slot, Tenant.
  3. Enable i18n (Settings → Internationalization) and add tenant relation + API filters.
  4. Configure roles & permissions and optional lifecycle hooks.
  5. Point mosc-temp at Strapi via env; implement tenant resolution and API calls.
